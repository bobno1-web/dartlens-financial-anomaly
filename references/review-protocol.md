# 리뷰 절차 (review-protocol)

> **목적:** `cr1` 서브에이전트를 **언제·어떻게** 쓰는지와 변경 작업 규율. cr1 정의 자체는 `.claude/agents/cr1.md`(수정하지 않음).

## cr1은 무엇인가
- 이 프로젝트 전용 코드 리뷰 서브에이전트. **리뷰·위험 보고만** 하고 **파일을 수정하지 않는다.**
- 도구: 읽기 전용 + Bash(테스트 실행). 코드/설계의 정확성·감사가능성·재현성·데이터 안전을 본다.

## 언제 cr1 리뷰를 받나 (주요 변경)
- OpenDART 데이터 로딩/수집 코드
- pandas/dataframe 변환(merge·groupby·pivot·reshape)
- 비율/벤치마크/이상탐지 로직, 임계값 변경
- Excel 출력 로직
- 설계 결정(계정 매핑, peer 풀 구성, 파일 쓰기 방식)

자명한 오탈자·문서 수정 등 사소한 변경은 리뷰가 필요 없다.

## 리뷰 포커스 6가지
1. **하드코딩 위험** — 회사/계정/파일 특정 로직
2. **데이터 손실 위험** — 조용한 drop·덮어쓰기·정밀도 손실
3. **소스 추적성** — 출력 → 원본 filing 역추적 가능 여부
4. **설명가능성(explainability)** — 플래그가 사람이 이해·감사 가능한지, 한 표본에 과적합 아닌지
5. **과적합(overfitting)** — 한 회사/한 샘플 파일에만 맞춘 로직
6. **감사가능성·재현성** — 같은 입력 → 같은 출력, 파라미터·타임스탬프 기록

## 작업 규율
- **승인 전 구현 금지.** `docs/`의 해당 계획이 승인된 뒤 비즈니스 로직을 작성한다.
- **작고 감사가능한 증분 변경**을 선호한다. 큰 변경은 리뷰가 어렵다.
- 리뷰는 **must-fix**(정확성·데이터안전·감사)와 **optional**(스타일·경미)을 구분해 보고한다.
- 테스트는 위험 경로를 지키는 **소수의 실용 테스트**만 요구한다(과도한 커버리지 요구 금지).

## 호출 방법
- Agent 도구에서 `subagent_type: "cr1"`으로 호출한다.
- 코드가 아직 없으면 계획/설계 텍스트를 리뷰 대상으로 전달할 수 있다(파일 미생성).
