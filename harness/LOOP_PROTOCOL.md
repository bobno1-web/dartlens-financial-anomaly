# LOOP_PROTOCOL — 랠프 루프 1회의 규율

> **목적:** 매 루프가 흔들리지 않도록 "한 번의 루프"를 절차로 고정한다.
> 이 파일은 절차의 정본이다. 방(개발/검증)의 세부 규율은 각각
> [`DEV_ROOM.md`](./DEV_ROOM.md) · [`VALIDATION_ROOM.md`](./VALIDATION_ROOM.md),
> 위반 불가 설계 규칙은 [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md)에 있다.
> 상위 프로젝트 규칙은 저장소 루트 `CLAUDE.md`가 정본이며, 이 문서는 그와 모순되지 않는다.

## 0. 핵심 원칙 (세 줄)

1. **한 루프는 한 가지 목적만 가진다.** 목적이 둘이면 루프를 쪼갠다.
2. **검토자(사람)가 방향을 정한다.** 개발방·검증방은 *다음 단계*를 스스로 정하지 않는다.
3. **완료조건을 먼저 못 박고 시작한다.** 완료조건 밖의 일은 이번 루프에서 하지 않는다.

## 1. 루프의 5단계

```
[검토자] 목적 1개 + 완료조건 1세트 확정
        │
        ▼
[개발방] 완료조건대로 실행 (조사 or 구현) ──► 보고(파일:라인 근거)
        │
        ▼
[검증방] 개발방 보고를 믿지 않고 독립 재현/재확인
        │
        ▼
        판정: PASS / PASS_WITH_NOTES / FAIL
        │
        ▼
[검토자] 판정·피드백을 해당 MD에 반영 → 다음 루프 목적 지정
```

각 단계의 산출물:

| 단계 | 주체 | 산출물 |
|---|---|---|
| 목적/완료조건 | 검토자 | 완료조건 체크박스 1세트 |
| 실행 | 개발방 | 항목별 결과 + 근거(파일:라인) |
| 검증 | 검증방 | 3종 판정 + (필요시) 수정 제안 |
| 반영 | 검토자 | MD 업데이트 지시, 다음 루프 지정 |

## 2. 완료조건 규칙

- 완료조건은 **루프 시작 시 확정**하고, 도중에 늘리지 않는다(스코프 크리프 금지).
- 완료조건은 **검증 가능**해야 한다("잘 됨" 금지 → "pytest 92 passed", "git diff --stat 0줄" 처럼).
- **체크박스가 전부 충족되기 전에는 어느 방도 "루프 종료"를 선언하지 않는다.**
- 조건 하나라도 미충족이면 그 루프는 **미완**이며, 검증 판정은 최대 `PASS_WITH_NOTES`가 아니라 `FAIL` 또는 "미완 보고"다.

## 3. 루프 유형

| 유형 | 개발방이 하는 일 | 하지 않는 일 |
|---|---|---|
| **조사 루프** | 읽기·추적·근거 수집·보고 | 코드/설정 수정, 커밋 |
| **구현 루프** | 완료조건에 명시된 최소 변경 | 완료조건 밖 변경, 리팩터링 끼워넣기 |
| **하네스/문서 루프** | 규율·문서·훅 파일 생성 | 제품 코드(src/*, app.py) 수정 |

유형은 검토자가 루프 시작 시 지정한다. 개발방이 임의로 바꾸지 않는다.

## 4. 커밋·병합 규율

- 생성/수정은 **워킹트리에만** 둔다. **커밋·병합·push는 검증 PASS 이후 검토자 지시로만** 한다.
- 조사 루프는 `git diff --stat`가 **0줄**이어야 정상이다.
- 파괴적 명령(대량 삭제·강제 덮어쓰기·history 수정)은 사전 확인 없이는 금지(→ `CLAUDE.md`, `references/safety-rules.md` §9).

## 5. 판정 3종 (검증방 정본은 VALIDATION_ROOM.md)

- **PASS** — 완료조건 전부 충족 + 독립 재현 성공 + 불변식 위반 없음.
- **PASS_WITH_NOTES** — 완료조건은 충족하나 후속에서 볼 개선점/주의가 있음. 진행 가능.
- **FAIL** — 완료조건 미충족, 재현 실패, 또는 [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md) 위반. 다음 루프로 못 넘어간다.

## 6. 피드백 반영

- 검증 판정과 검토자 코멘트는 **말로 흘리지 않고 해당 MD에 반영**한다.
  - 절차 관련 → 이 파일 / `DEV_ROOM.md` / `VALIDATION_ROOM.md`
  - 설계 불변식 관련 → `DESIGN_INVARIANTS.md`
  - 제품 결정 → 저장소 `docs/DECISIONS.md`
- 반영은 **추가·수정**이며, 과거 근거를 조용히 지우지 않는다(감사가능성).

## 7. "스스로 다음 단계 정하지 않기"의 의미

- 개발방/검증방의 recap 끝에 "다음은 X를 하겠다"를 **쓰지 않는다.** 대신 "다음 후보: X(검토자 판단 요청)"까지만.
- 다음 루프의 목적·유형·완료조건은 **검토자만** 지정한다.
- 이는 자율성 제한이 아니라, 감사가능성과 방향 통제를 위한 규율이다.

## 8. 이 문서의 위치

- 정본 계층: `CLAUDE.md`(프로젝트 절대규칙) ⊃ `references/*`(상세) ⊃ 이 `harness/*`(루프 운영 규율).
- 셋이 모순되면 **상위(CLAUDE.md/references)가 이긴다.** 모순 발견 시 개발방/검증방은 임의 조정하지 말고 **보고**한다.
