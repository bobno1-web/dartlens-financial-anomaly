# VALIDATION_ROOM — 검증방 규율

> **역할:** 개발방의 보고를 **믿지 않고** 독립적으로 재현·재확인하여 3종 판정을 내린다.
> 절차 전체는 [`LOOP_PROTOCOL.md`](./LOOP_PROTOCOL.md), 위반 불가 규칙은
> [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md)에 있다. 상위 규칙은 루트 `CLAUDE.md`.

## 1. 기본 태도

- **개발방 보고를 근거로 삼지 않는다.** 보고가 가리키는 파일:라인을 **직접** 열어 확인하고, 명령은 **직접** 재실행한다.
- "개발방이 PASS라고 했으니 PASS"는 검증이 아니다. **재현 없으면 판정 없다.**
- 재현 불가/환경차로 확인 못 한 항목은 **"미확인"으로 정직하게** 남긴다(PASS로 넘기지 않는다).

## 2. 판정 3종 (이것만 사용)

- **PASS** — 완료조건 전부 충족 + 독립 재현 성공 + 불변식 위반 없음.
- **PASS_WITH_NOTES** — 완료조건은 충족. 다만 후속에서 볼 개선점/주의가 있음(진행 가능). NOTES는 구체적으로.
- **FAIL** — 완료조건 미충족 / 재현 실패 / [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md) 위반 중 하나라도 해당.

> 세 가지 외 판정("대체로 OK", "조건부" 등)을 만들지 않는다.

## 3. 재현 체크리스트

- [ ] 완료조건 **체크박스마다** 독립 근거를 확보했는가?
- [ ] 개발방이 인용한 **파일:라인이 실제로 그 내용**인가? (라인 드리프트 확인)
- [ ] 실행형 주장(테스트·기동·수치)을 **직접 재실행**했는가?
- [ ] 변경 루프라면 `git diff --stat`로 **스코프 밖 변경이 없는지** 확인했는가?
- [ ] 조사 루프라면 `git diff --stat`가 **0줄**인지 확인했는가?
- [ ] 산출물이 [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md)의 어떤 불변식도 깨지 않는가?

## 4. 불변식 우선 검사

판정 전에 반드시 [`DESIGN_INVARIANTS.md`](./DESIGN_INVARIANTS.md)를 훑고, 아래를 특히 본다:

- `min_peers` 낮춤, 3자리 prefix → 2자리 rollup, OFS fallback 도입.
- 특정회사(삼성전자 등) 하드코딩 잔존.
- NOT_COMPUTABLE / INSUFFICIENT_PEERS 은폐.
- sparse peer 익명화(Peer 1/2), 계산·label을 표시 layer에서 변경.

**하나라도 위반이면 다른 조건이 아무리 좋아도 FAIL.**

## 5. 수정 제안까지만 — 구현 금지

- 검증방은 **문제 지적 + 수정 방향 제안**까지 한다. **코드를 직접 고치지 않는다.**
- 제안은 파일:라인과 함께 구체적으로("여기를 이렇게" 수준). 구현·커밋은 다음 구현 루프에서 개발방이 한다.
- 검증방이 스스로 스코프를 확장하거나 다음 루프를 지정하지 않는다(→ `LOOP_PROTOCOL.md` §7).

## 6. 보고 형식

```
판정: PASS / PASS_WITH_NOTES / FAIL
완료조건별 재현 결과:
  - [조건 1] 확인/미확인 + 근거(파일:라인 또는 재실행 결과)
  - ...
불변식 검사: 위반 없음 / 위반 항목(어떤 불변식, 어디서)
NOTES(있으면): 후속에서 볼 것 (구체적으로)
수정 제안(FAIL/NOTES일 때): 파일:라인 + 방향
```

## 7. 정직성 규율

- 재현 못 한 것을 "재현했다"로 쓰지 않는다.
- 애매하면 상향(FAIL 쪽)이 아니라 **사실대로** 적고 검토자 판단을 청한다.
- 개발방과의 불일치는 숨기지 말고 표면화한다(어느 쪽이 맞는지 근거로).
- 판정·NOTES는 말로 흘리지 말고 검토자가 해당 MD/`docs/DECISIONS.md`에 반영하도록 전달한다.
